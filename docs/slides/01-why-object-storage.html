Why Build a Database on S3?

Search engines inherited an expensive body plan from relational databases. Zeppelin — like turbopuffer before it — starts from a different premise: keep the data where storage is nearly free, and spend engineering effort clawing back the latency.

The economics

A conventional search or vector database keeps its data on replicated local disks, with the hot set pinned in RAM. That architecture was designed for transactional workloads that need sub-millisecond writes. Search doesn’t: its write pattern looks like a data warehouse (big batches, no transactions), while its reads just need to land under a couple hundred milliseconds.

The cost difference between those two body plans is not subtle. In turbopuffer’s launch essay — the founding text of this architecture, and Zeppelin’s explicitly documented reference point — the first-principles math per terabyte-month works out roughly to:

Storage tier~$/TB/monthUsed by
RAM + triply-replicated SSD$3,600classic in-memory vector DBs
Triply-replicated SSD$600disk-based search engines
Object storage + SSD cache$70turbopuffer, Zeppelin
Raw object storage$20the cold floor

Zeppelin’s own internal cost sketch reaches the same conclusion from the bottom up: 10 million 768-dimensional vectors occupy about $0.72/month of S3 storage. At that point storage is a rounding error — the entire bill is compute, and compute can be stateless, shared, and scaled to zero.

What S3 gives you for free

  • Durability. Eleven nines, cross-AZ replication, no RAID to babysit. A write acknowledged by S3 is done — there is no separate replication protocol to design or break.
  • Stateless nodes. If every byte of truth lives in the bucket, any node can serve any namespace. Nodes crash, restart, scale out — nothing is lost, nothing needs rebalancing. A local disk is purely a cache, and (per the project’s own design rules) “local cache is disposable.”
  • Per-tenant isolation by prefix. Each namespace is just an S3 key prefix. Creating a tenant costs one small object; a million idle tenants cost storage, not servers.
  • A concurrency primitive. Since November 2024, S3 supports conditional writes (If-Match on an object’s ETag). That single feature lets S3 replace an entire coordination service — Chapter 03 is about exactly this.
Jargon: ETag
An opaque version identifier S3 attaches to every object. “Write this object, but only if the ETag still equals X” is a compare-and-swap (CAS): it succeeds for exactly one of any set of concurrent writers, and the losers find out.

The tax: every read is a network hop

The price of this architecture is latency. An S3 GET takes roughly 25–30 ms (Zeppelin’s benchmarks measured ~28 ms per GET, ~44 ms per PUT). A query that naively touched ten objects in sequence would take 300 ms before doing any math. The whole design of an S3-native engine is therefore organized around one currency: roundtrips.

Three consequences of the ~28 ms tax shape everything in the chapters ahead:

  1. Index structures are chosen by roundtrip count, not raw speed. Zeppelin uses centroid-based indexes that answer a cold query in 2–3 sequential GETs, and avoids graph indexes (HNSW and friends) that would hop object-to-object dozens of times (Chapter 05).
  2. Caching is not an optimization; it is the product. A memory-and-disk cache in front of S3 turns 400 ms cold queries into ~10–100 ms warm ones (Chapter 04). turbopuffer reports the same shape: ~874 ms cold p50 → ~14 ms warm at 1M documents.
  3. Fetches are batched and parallelized ruthlessly. The single biggest speedup in Zeppelin’s benchmark history — 615 ms to 132 ms per query — came from issuing S3 GETs concurrently instead of sequentially (Chapter 10).

Where the idea comes from

Zeppelin did not invent this thesis and doesn’t claim to. Its own gap-analysis notes state it plainly: “Zeppelin and turbopuffer share the same core thesis: S3-native vector search where object storage is the source of truth, nodes are stateless, and you pay storage prices, not RAM prices. turbopuffer is the production leader in this space (powering Cursor, Notion, etc.).” The same family includes Chroma’s wal3 log (Chapter 03) and, more distantly, every LSM-tree database since BigTable — writes land in a log, background work folds the log into immutable, indexed segments.

What Zeppelin adds to the family is chiefly its verification culture — model-checking the concurrency design in TLA+ before and after implementing it (Chapter 09) — and the fact that the whole thing is Apache-2.0 and self-hostable.
